The lenses

Health, measured through ten lenses.

Each lens is a deterministic measurement of the code as it stands — never a judgement of the team. The CAI groups roughly 120 dimensions under these lenses; each dimension is computed by a tool reading your code, and the lenses are how those dimensions roll up into one reproducible 0–100 number.

What each lens measures.

The five core lenses are always on — they apply to any C#/.NET codebase. The remaining lenses are model-aware: they light up only when the architecture calls for them, and the weights re-normalise so a repository is never penalised for a lens that doesn't apply.

LensWhat it measuresWhat it protects against
Code Health core The intrinsic state of the source: method complexity and branching, duplication, oversized classes, dead scaffolding, and the presence and reach of tests. Code that's expensive and risky to change — tangled methods, copy-pasted logic, and untested paths that turn small edits into regressions.
Architecture core How the system is structured: layering, dependency direction and cycles, the integrity of bounded contexts, and hidden change-coupling mined from history. Structural decay — leaking boundaries, dependency cycles, and modules that quietly move together until the design can no longer be reasoned about a piece at a time.
Maturity core How the codebase has evolved over time: churn-versus-complexity hotspots, knowledge concentration (bus factor), and knowledge freshness — read from the git history. Latent maintenance risk a snapshot can't see — key-person dependencies, orphaned modules, and the hotspots where cleanup pays back most.
Production-Readiness core Whether the code is fit to run and operate: deployment automation, error handling, observability, and operational safeguards such as idempotency on consumers. Failures that only surface in production — swallowed exceptions, silent outages, manual deploys, and non-idempotent handlers that corrupt state under retry.
Security & Compliance core Posture against known risk: vulnerable dependencies, committed secrets, missing injection guards, and the automatable subset of common controls. Exploitable exposure — live CVEs, leaked credentials, and unmet baseline controls that turn a codebase into an incident waiting to happen.
Domain Modelling model-aware Where the code expresses a rich domain model, the fidelity of that model: aggregate integrity, invariants, and the discipline of the ubiquitous language. An anaemic or leaking domain — business rules scattered outside the model, where the design no longer protects its own invariants.
Event-Driven model-aware Where messaging or events are in play, the soundness of the asynchronous design: contracts, delivery semantics, and handler robustness. Brittle integration — lost or duplicated messages, unhandled poison events, and consumers that can't survive redelivery.
Event Sourcing model-aware Where state is rebuilt from an event log, the integrity of that approach: event and stream design, versioning, and projection consistency. An unrecoverable log — un-versioned events and divergent projections that make the system's source of truth impossible to replay safely.
Accessibility model-aware Where the code renders a user interface, conformance signals for accessible markup and interaction that can be measured from the source. Interfaces that exclude users — and the regulatory and reputational exposure of shipping a UI that doesn't meet baseline accessibility expectations.
Performance model-aware Where the architecture has performance-sensitive paths, statically detectable patterns that bear on throughput and latency, such as inefficient data access. Designed-in slowness — N+1 queries and hot-path inefficiencies that hold up under a demo and fall over under load.

Together these are the ten lenses. Which model-aware lenses apply is decided by the architecture itself, not by configuration — so the survey measures only what the codebase actually is.

Evidence, not opinion.

Some of the most predictive signals aren't in any single file — they're in how the code has changed over time. The CAI mines them deterministically from your git history and folds them into the same number through the Maturity and Architecture lenses:

Churn × complexity hotspots

Files that change often and carry high complexity — the most predictive maintenance-risk signal there is, located rather than guessed at. Feeds Maturity.

Bus factor

Files whose living knowledge sits with a single author, by time-decayed authorship — key-person risk, pinned to specific code. Feeds Maturity.

Hidden coupling

Files that keep changing together across a boundary with no code dependency — logical coupling static analysis is blind to. Feeds Architecture.

These are computed by tool, never by an LLM. And where the history is shallow or absent, they abstain rather than guess — a missing measurement is reported as missing, never invented.

Always free to compute the full set.

Every tier computes all ten lenses and every applicable dimension. Depth is never paywalled: the same commit re-scores to the same number whoever runs it, and the full lens breakdown is part of every survey. What higher tiers add is breadth — portfolio roll-ups, team analytics, cadence — never a deeper look at any one codebase.

See the rule behind every number.

Each lens decomposes into dimensions, and each dimension is scored by a rule you can read. The specification defines the lenses, the frozen rubric, and the bands; the CLI lets you compute the full set yourself and confirm it lands on the same number.
