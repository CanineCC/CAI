Specification · CAI-1.0

The CAI algorithm

The Codebase Assurance Index is one reproducible 0–100 score for a whole C#/.NET codebase. It is computed deterministically from evidence under a frozen, versioned rubric — a measurement, not an opinion. Same code in, same rubric, same number out.

Inputs → score

The CAI is the output of a fixed pipeline. Nothing in it is a free-form opinion; every stage is defined by the rubric and reproducible by anyone holding the same inputs.

1 · Evidence

A static analyzer reads the source and emits deterministic findings. Git history is mined in the same pass for behavioural signals. This is observed fact about the code — not a model's judgement.

2 · Dimensions

Evidence is scored into roughly 120 named dimensions, each with a stable ID (e.g. D5) and a published rule. Each dimension is one tool measurement.

3 · Lenses

Dimensions are grouped into lenses — Code Health, Architecture, Maturity, and so on — the unit a contract pins against.

4 · Weighted roll-up

Lens scores are combined under the rubric's weights into a single 0–100, then placed in a band. Inapplicable lenses drop out and the weights re-normalise.

An advisory LLM read may annotate a small number of findings in plain English. By construction it sits outside this pipeline and can never move the headline number.

The frozen rubric is versioned

A score is meaningless without the ruler it was measured against. Every CAI is therefore reported together with the rubric version it was computed under — for example rubric-2026.08.15.

The core property of the standard is reproducibility: the same commit, scored under the same rubric version with the same advisory data, yields byte-for-byte the same number. Any change to a scoring rule that could move the score for unchanged code requires a rubric version bump — so a number is always re-derivable from a rule you can read.

CAI = f(commit, rubric_version)

f(commit, rubric-2026.08.15) → 73   # today
f(commit, rubric-2026.08.15) → 73   # next year, same inputs, same output

Lenses & dimensions

The CAI is composed of approximately 120 dimensions grouped into lenses. Five core lenses are always on, because they apply to any C#/.NET codebase:

Code Health

Complexity, duplication, oversized types, dead code, test depth.

Architecture

Layering, dependency cycles, bounded-context leakage, change coupling.

Maturity

Evolution over time, hotspots, knowledge concentration and freshness.

Production-Readiness

Deployment automation, error handling, observability, idempotency.

Security & Compliance

Known-vulnerable dependencies, committed secrets, injection guards, automatable controls.

Model-aware lenses — Domain Modelling, Event-Driven, Event Sourcing, Accessibility, Performance — light up only when the architecture calls for them. A codebase is never penalised for a lens that does not apply, and a single lens reading Critical caps the headline so a serious failure cannot hide behind strong scores elsewhere.

We never score a line

Some of the most predictive signals are not in any single file — they are in how the code has changed over time. These are mined deterministically from git history and folded into the same number, through the Maturity and Architecture lenses:

Churn × complexity hotspots

Files that change often and carry high complexity — the strongest maintenance-risk signal there is, over a rolling window.

Hidden coupling

Files that keep changing together across a module boundary with no code dependency — logical coupling static analysis is blind to.

Bus factor / key-person

Code whose living knowledge sits with a single author, by time-decayed authorship — key-person risk located, not guessed.

Where history is shallow, these dimensions abstain rather than guess. No history, no measurement: an absent or shallow git log makes the affected dimensions report not measured and drop out of the roll-up — they are never filled with a fabricated value.

The standard evidences the automatable slice of codebase quality. It never certifies, and it never scores a line of code by reading it like a reviewer — every dimension is a rule over evidence.

The bands

A raw 0–100 is placed into one of four bands. Bands are the human-readable summary; the underlying number is always available.

Strong80–100
Fair60–79
Weak40–59
Poor0–39

A contract floor of CAI ≥ 80 (Strong) means every always-on lens is Strong or better with no lens Critical — decomposable, not opaque.

Versioning policy

The specification itself is versioned (this document is CAI-1.0), and so is the rubric. The two move independently: the spec defines the algorithm; the rubric instantiates the rules and weights the algorithm runs against.

Frozen for a contract

An engagement can pin a repository to a frozen rubric version for the duration of a contract, so bidders price against fixed criteria. The ruler stops moving under the parties — any movement in the score is the asset changing, never the rubric.

Bumps are disclosed

Any rule change that can move a score for unchanged code bumps the rubric version and is recorded in the changelog. A scoring change not reflected in the published spec fails CI.

Advisory data refreshes

Vulnerability data refreshes even under a frozen rubric, so a newly published CVE can legitimately move a security finding. That is a real signal, disclosed — not a moved ruler.

Deviations are disclosed

Where a survey applies a profile, suppression, or compliance declaration, it changes what the artifact says, never the CAI — and the deviation is stated on the face of the report.

Don't take our word for it

The algorithm is open and the score is reproducible. Run it yourself, or check a survey you've been handed against the published rubric.
